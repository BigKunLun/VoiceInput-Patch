# 闪电说 → Claude Code 语音输入补丁 - 调试记录

## 目标

解决闪电说通过「剪贴板 + Cmd+V」粘贴文本时，Claude Code 检测到多行粘贴会折叠的问题。

---

## 第一轮尝试（已失败）

### 方案一：剪贴板轮询 + 清空抢跑（v2）

**原理**：轮询剪贴板变化 → 清空剪贴板 → 等闪电说的 Cmd+V 粘贴空内容 → CGEvent 逐字输入

**结果**：时序竞争失败。闪电说写入剪贴板和发 Cmd+V 几乎同时，60ms 轮询抢不赢。

### 方案二~五：CGEvent Tap 拦截 Cmd+V + 各种打字方式

**原理**：用 CGEvent Tap 拦截 Cmd+V → 吞掉粘贴 → 改用 type_unicode 逐字输入

**结果**：Cmd+V 拦截成功，但 type_unicode 打字失败。尝试了同进程打字、外部子进程打字、信号暂停 tap 等方式，均失败。

---

## 深入研究（根因分析）

### 1. CGEvent Tap 干扰 type_unicode 的根因

**核心发现：`.defaultTap` 是 WindowServer 事件管道中的同步阻塞节点。**

macOS 事件管道流程：
```
硬件/CGEvent.post(.cghidEventTap)
  → IOKit HID
  → WindowServer
  → cgSessionEventTap         ← 我们的 tap 在这里
  → cgAnnotatedSessionEventTap
  → 目标应用（Ghostty）
```

- type_unicode 用 `.cghidEventTap` 发事件，事件从管道顶部进入，必须**同步流经**所有 active tap
- 即使 callback 只是检查 keyCode 后立刻 return，每个 keyDown 仍需完成一次 **Mach IPC 往返**
- type_unicode 发射频率 ~1250Hz（800μs/字符），加上 keyDown+keyUp = ~2500 events/sec
- 高频 IPC 开销累积 → 事件队列拥塞 → 事件丢失（实测 8 字只出来 2 字）

**为什么 `tapEnable(false)` 没用：**
- 仅禁用不够，Mach port 仍注册在 WindowServer 中
- 要彻底消除影响需要四步：`tapEnable(false)` → `CFMachPortInvalidate` → `CFRunLoopRemoveSource` → `CFRelease`
- 信号方式有时序竞争（signal handler 和 RunLoop 不在同一线程，无内存屏障）

### 2. Claude Code 粘贴折叠机制

**Bracket Paste Mode**：
- Claude Code 自己发送 `\e[?2004h` 启用 bracket paste mode
- Ghostty 收到 Cmd+V 时在粘贴内容前后添加 `\e[200~` / `\e[201~`
- Claude Code 检测到这些序列 → 判定为粘贴 → 折叠显示
- **外部无法覆盖** Claude Code 自己启用的 bracket paste
- Ghostty 没有禁用 bracket paste 的配置

**Claude Code 没有禁用折叠的设置**。相关 Issues：
- [#3412](https://github.com/anthropics/claude-code/issues/3412) - 164 个赞，已分配给 Anthropic 员工
- [#23134](https://github.com/anthropics/claude-code/issues/23134) - 请求禁用选项
- [#23702](https://github.com/anthropics/claude-code/issues/23702) - 请求可配置阈值

### 3. 闪电说 AA Bug 分析

**AA bug 原因**：
- CGEvent 的 virtual keycode 0 = 'a' 键
- `keyboardSetUnicodeString` 只是"覆盖" Unicode 字符串，某些应用可能忽略覆盖转而读 keycode
- Apple 文档明确警告："application frameworks may ignore the Unicode string and do their own translation based on the virtual keycode"
- AA（两个大写 A）暗示 keyDown + keyUp 各产生一个字符，且 Shift 状态被意外继承

## 最终实施方案（已验证）

### 核心改动

基于方案 A（改 post 位置）+ 额外修复，共三处关键改动：

#### 1. type_unicode.swift：post 位置改为 `.cgAnnotatedSessionEventTap`

```swift
// 改前
keyDown.post(tap: .cghidEventTap)
// 改后
keyDown.post(tap: .cgAnnotatedSessionEventTap)
```

事件从管道下游注入，**跳过 cgSessionEventTap 上的 CGEvent Tap**，避免干扰。

**验证结果**：CGEvent Tap 运行时发送"有tap也能完整输出十个字" → 全部出来 ✅（之前只出来 2/8）

#### 2. type_unicode.swift：显式清除修饰键标记 `flags = []`

```swift
keyDown.flags = []
keyUp.flags = []
```

防止 `CGEventSource(.hidSystemState)` 继承系统当前的修饰键状态（如闪电说 Cmd+V 的 Cmd 还没松开时，事件会带 Cmd 标记 → Cmd+A 全选等快捷键被触发）。

**注意**：曾尝试用 `.privateState` 替代 `.hidSystemState`，但 `.privateState` + `.cgAnnotatedSessionEventTap` 组合会丢字，放弃。

#### 3. voice_type.py：换行替换为空格

```python
text = text.replace("\n", " ")
```

闪电说的语音文本可能包含换行，type_unicode 会把 `\n` 当 Return 键发送，在 Claude Code 中会提前提交消息。

### 架构

```
闪电说 → 写入剪贴板 + Cmd+V
                              ↓
intercept_paste (Swift)  →  CGEvent Tap 拦截 Cmd+V → 返回 nil 吞掉
                              → 读取剪贴板 → base64 编码 → stdout
                              ↓
voice_type.py (Python)   →  读 stdout → 换行替换空格 → 等 200ms Cmd 释放
                              → 调用 type_unicode
                              ↓
type_unicode (Swift)     →  CGEvent + keyboardSetUnicodeString
                              → flags = [] 清除修饰键
                              → post 到 .cgAnnotatedSessionEventTap（绕过 tap）
                              ↓
Ghostty/Claude Code      →  收到逐字键盘输入（非粘贴）→ 不触发折叠
```

### 启动方式

```bash
python3 ~/CodingTime/Personal/VoiceInput-Patch/voice_type.py --whitelist Ghostty
```
